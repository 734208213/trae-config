## 编码规范

### 命名约定
- 使用描述性、意图明确的命名
- 布尔值优先使用 `isXxx`、`hasXxx`、`canXxx` 前缀
- 函数使用动词+名词结构，如 `getUserInfo`

### 注释规范
- 注释解释"为什么"而非"是什么"
- 删除死代码，不要用注释屏蔽代码
- 复杂逻辑必须添加注释说明意图
- 公共 API 必须添加 JSDoc 注释

### 代码格式
- 遵循项目现有代码风格
- 缩进统一使用 4 个空格
- 使用 Prettier/ESLint 等工具统一格式

### 导入顺序
- 分组：标准库 → 第三方 → 本地模块
- 组内按字母顺序排列，组间空一行

### 错误处理
- 快速失败，提供描述性错误信息
- 绝不静默吞掉错误
- 记录足够的上下文信息便于调试

## 工作流原则

### 先搜索后编码
- 修改前充分理解代码库
- 查看现有实现再决定方案
- 避免重复造轮子

### 小步提交
- 相关变更分组提交
- 编写有意义的提交信息

### 验证后再完成
- 运行测试确保通过
- 运行 lint 和 typecheck 命令

### 持续学习
- 从会话中提取可复用模式
- 积累项目特定知识

## IDE 特性利用

- 利用 LSP 智能提示和跳转定义
- 使用查找引用了解影响范围
- 使用内置重构功能
- 添加有意义的断点调试
